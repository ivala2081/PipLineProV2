# Internationalization (i18n)

**Status:** Living spec · reflects codebase as of `main` on 2026-04-24
**Scope:** Language switching, string externalization, locale-aware formatting
**Related:** [design-system/accessibility.md §5](./design-system/accessibility.md#5-internationalization), [design-system/patterns.md §12](./design-system/patterns.md#12-currency--locale)

> Two locales ship today: English (`en`) and Turkish (`tr`). Turkish is the primary production locale. Turkish strings are ~30–40% longer than English — design for that.

---

## 1. Stack

| Tool | Role |
|---|---|
| `i18next` | Core engine |
| `react-i18next` | React bindings |
| `i18next-browser-languagedetector` | Detects user's preferred language |
| `i18next-http-backend` (optional, not used) | — |

Locale files ship as static JSON in `src/locales/` (not lazy-loaded — both locales bundled upfront).

## 2. File layout

```
src/locales/
├── en/
│   ├── common.json         # Generic verbs / UI words
│   ├── components.json     # Strings inside @ds components
│   └── pages.json          # Feature-specific strings
└── tr/
    ├── common.json
    ├── components.json
    └── pages.json
```

Setup: [src/lib/i18n.ts](../src/lib/i18n.ts). Provider wraps the app in `main.tsx`.

## 3. Namespace conventions

| Namespace | Scope | Examples |
|---|---|---|
| `common` | Verbs, universal UI words | `common:save`, `common:cancel`, `common:loading`, `common:delete` |
| `components` | Strings inside `@ds` components | `components:dialog.close`, `components:pagination.previous`, `components:datePicker.today` |
| `pages` | Feature-specific strings | `pages:transfers.title`, `pages:accounting.tabs.ledger`, `pages:hr.bonuses.autoLabel` |

**Rule:** feature code uses `useTranslation('pages')` by default. Design system components use `useTranslation('components')`. Verbs-only use cases use `useTranslation('common')`.

## 4. Key naming

- Dot-separated, feature-scoped: `transfers.filters.employee`, `accounting.settings.adminOnly`.
- Lowercase-camelCase within each segment.
- First segment is the feature/page name for `pages`, or the component name for `components`.
- Avoid string interpolation in the key itself — use `{{variable}}` values instead.

## 5. Detection and fallback

```
localStorage (i18nextLng)  →  navigator.language  →  'en'
```

Once the user manually picks a language, it's persisted to `localStorage` and used on subsequent visits. The detector only runs on first load.

## 6. Locale-aware formatting

### 6.1 Numbers

```ts
n.toLocaleString('tr-TR', { minimumFractionDigits: 2, maximumFractionDigits: 2 })
n.toLocaleString('en-US', ...)
```

Canonical helpers:
- `formatNumber` in [src/pages/transfers/transfersTableUtils.ts:80–86](../src/pages/transfers/transfersTableUtils.ts#L80-L86) — money.
- `fmtNum` / `fmtAmount` in [src/pages/hr/utils/salaryCalculations.ts](../src/pages/hr/utils/salaryCalculations.ts) — salary amounts.

When used in a data-dense cell, pair with `font-mono tabular-nums` for digit alignment. See [design-system/patterns.md §10](./design-system/patterns.md#10-numeric-formatting).

### 6.2 Dates

```ts
date.toLocaleDateString('tr-TR', { day: 'numeric', month: 'long', year: 'numeric', weekday: 'long' })
```

Feature-specific formatters in [src/lib/date.ts](../src/lib/date.ts): `localYMD`, `localDayStart`, `localDayEnd` — timezone-aware (Europe/Istanbul for the default org).

### 6.3 Currency symbols

Rendered inline as `₺` / `$` — **not** via `Intl.NumberFormat currency` style (we prefer explicit symbol control for the data-density aesthetic).

## 7. Interpolation

Use `{{variable}}` notation — never string concatenation:

```tsx
// yes
t('transfers.count', { count })
// returns: "5 transfers" (en) / "5 transfer" (tr)

// no
`${t('transfers.countBefore')} ${count} ${t('transfers.countAfter')}`
```

**Pluralization:** `i18next` handles this when you pass `count` and provide `_plural` variants in the JSON:

```json
{
  "transfers_one": "{{count}} transfer",
  "transfers_other": "{{count}} transfers"
}
```

Turkish doesn't need plural variants (no grammatical plural change); English does.

## 8. String-length budget

Turkish is 30–40% longer than English on average. Examples from the app:

| English | Turkish | Δ |
|---|---|---|
| "Save" | "Kaydet" | +50% |
| "Cancel" | "İptal" | +25% |
| "Add Transfer" | "Transfer Ekle" | +15% |
| "Withdrawal" | "Çekme" | −40% (opposite — rare) |
| "Settings" | "Ayarlar" | +15% |

**Design rules** (lifted from [design-system/accessibility.md §5.3](./design-system/accessibility.md#53-string-length)):

- Never set fixed pixel width on elements containing translated text.
- Buttons must allow label wrap or tooltip-on-hover truncate.
- Tables can truncate cells but expose the full value via `title` attr.
- Form labels stack above inputs, never to the left (Turkish labels often exceed left-column width).

## 9. `useLocale()` hook

[src/design-system/hooks/useLocale.ts](../src/design-system/hooks/useLocale.ts):

```ts
const { locale, changeLocale, locales, localeNames } = useLocale()
// locale:       'en' | 'tr'
// changeLocale: setter (persists to localStorage)
// locales:      ['en', 'tr']
// localeNames:  { en: 'English', tr: 'Türkçe' }
```

Used by the language picker in the app header. Feature code usually reads `i18n.language` directly via `useTranslation().i18n.language`.

## 10. RTL readiness

We don't ship Arabic / Hebrew / Farsi today. But:

- Prefer logical properties where cheap (`ms-*` / `me-*` over `ml-*` / `mr-*`).
- Use `text-start` / `text-end` instead of `text-left` / `text-right` for symmetric elements.

This keeps future RTL cheap. Don't retrofit existing code en masse.

## 11. Adding a new locale

1. Create `src/locales/<lang>/{common,components,pages}.json` with full translations.
2. Add the locale code to `SUPPORTED_LANGUAGES` in [src/lib/i18n.ts](../src/lib/i18n.ts).
3. Add a display name to `LOCALE_NAMES` in [useLocale.ts](../src/design-system/hooks/useLocale.ts).
4. Verify that the language picker shows the new option.
5. Test long-string edge cases (filter bars, dialog footers).

## 12. Known gaps

- **No missing-key reporting.** i18next silently returns the key when a translation is missing. In DEV mode, it should console.warn; verify the config.
- **No contract tests.** Nothing ensures TR has every key EN has (or vice versa). A CI step comparing key sets would catch drift.
- **Plurals are under-used.** Many places use `{count} transfer` (no plural form at all). Refactor opportunistically.
- **Hardcoded Turkish in bulk payment descriptions.** Several places in [src/hooks/queries/useHrQuery.ts](../src/hooks/queries/useHrQuery.ts) write Turkish strings directly into `accounting_entries.description` (e.g. `'Toplu Maaş Ödemesi — …'`). Move to i18n if we ship non-TR orgs.
- **No locale-aware currency in UI.** We render `₺` and `$` hardcoded. If we onboard an EUR org, we'd need a `currencySymbol(currency)` helper.
- **Date picker formats via `toLocaleDateString(undefined, ...)`** in some places ([DatePicker.tsx:155–163](../src/design-system/components/DatePicker/DatePicker.tsx#L155-L163)). Undefined locale means "user's browser locale" — might not match `i18n.language`. Pass `locale === 'tr' ? 'tr-TR' : 'en-US'` explicitly.
- **No fallback key strategy.** When a key is missing, we get the raw key string. Some UIs could provide English fallbacks from the `en` bundle when the current locale lacks the key. Configure `fallbackLng: 'en'` in i18next.
